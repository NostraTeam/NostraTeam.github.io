The following tree shows the directory structure of a Nostra project, and the paragraphs after the tree will explain the contents of each file and directory.

Note that some of the names in the directory tree are not always the same (e.g. because the file name contains the name of the project). In such a case, the name of the file in the tree will contain something like this: <project name>. The paragraph of the specific file will then contain further explanations what the file name actually will be.

Also, to enhance readability, in the following tree, regular file names are written in a regular font and directory names are written in a bold font. In addition to that, there are optional files. These are written in italics and their names are surrounded by round braces.

│   ├── CPackConfig.cmake.in
│   ├── Nostra<project name>Config.cmake.in

File Details

/

The root of the project. All files that are related to the project are in this directory of a subdirectory of it.

Note that / does not mean that the directory is the system's root directory - it is just a placeholder for the directory that the project files are in (it does not matter what that directory is called).

/cmake

A directory that contains files that are used by CMake. As of now, the files in this directory are exclusively files that will be configured by CMake using configure_file() .

/cmake/config.h.in

optional

A file that will be configured by CMake. The configured file can be included using
#include "nostra/<project name>/config.hpp
and it is meant to be used to inject variables from CMake into the C/C++ code.

To inject a CMake variable, a new macro needs to be created in this file:
#define MY_MACRO @<CMake variable name>@
where <CMake variable name> is the name of the variable that should be injected. Note that the name of the macro (MY_MACRO in this case) can be freely chosen - it is however common practice to make it an internal macro (thus making it start with <project prefix>_INTERNAL_) and then choose the rest of the name according to the CMake variable that will be injected.

Example: If a project has the prefix ntp and the injected variable is CMAKE_SOURCE_DIR, then a good name for the macro would be NTP_INTERNAL_SOURCE_DIR.

The default file that is generated by the CreateProject.cmake script creates a config.h.in that will inject the following variables:

CMake variable Macro name
PROJECT_VERSION_MAJOR <project prefix>_INTERNAL_VERSION_MAJOR
PROJECT_VERSION_MINOR <project prefix>_INTERNAL_VERSION_MINOR
PROJECT_VERSION_PATCH <project prefix>_INTERNAL_VERSION_PATCH
PROJECT_VERSION_TWEAK <project prefix>_INTERNAL_VERSION_TWEAK

/cmake/CPackConfig.cmake.in

A file that will be configured by CMake. It will be used to configure CPack and it contains a collection of set statements.

/cmake/Nostra<project name>.cmake.in

A file that will be configured by CMake. By default, it only includes the default configuration file generated by CMake.

It is possible to read more about the usage of this file here.

<project name> is the name of the project in upper camel case.

Example: For a project called TestProject, the filename would be NostraTestProjectConfig.cmake.in.

/cmake/welcome.txt.in

A file that will be configured by CMake. This file can be used by CPack installers to welcome a user.

/doc

A directory that contains files that are used by Doxygen to generate the documentation for the project.

/doc/dot

A directory that contains dotfiles that can be included into a Doxygen comment using \dot.

/doc/img

A directory that contains image files that can be included into a Doxygen comment using \image.

additional_documentation.dox

A file that contains additional documentation that can/should not be put into regular source files. These include additional pages (\page) or, for example, the documentation of namespaces.

/doc/Doxyfile.in

The template for the Doxyfile used by Doxygen that will be configured by CMake (to inject e.g. the project name).

Usually, the default Doxyfile.in should not be modified.

/doc/DoxygenLayout.xml

The DoxygenLayout.xml file used by Doxygen.

Usually, the default DoxygenLayout.xml should not be modified.

/doc/style.css

A CSS file that can be used to style the output of Doxygen.

/examples

A directory that contains the examples that are provided by the library.

/examples/<example name>.ex.d

A directory that contains the source file of one particular examples.

Note that there is a directory for each example. Thus, this chapter may describe more than one directory.

This page provides more information on how to write examples and what <example name> stands for in this case.

/examples/<example name>.ex.d/<example name>.ex.cpp

The source file of the example with the name <example name>.

Note that there is a source file for each example. Thus, this chapter may describe more than one file.

This page provides more information on how to write examples and what <example name> stands for in this case.

/include, /include/nostra and /include/nostra/<project name>

A directory that contains the header files of the project.

<project name> is the name of the project in all lowercase letters and with no spaces.

Example: For a project called TestProject, the name would be testproject.

/include/nostra/<project name>/<component name>.cpp

The header file of a single component.

Note that there is a header file for each component. Thus, this chapter may describe more than one file.

<component name> is the name of the component that the header belongs to (in the underscore separator notation).

/src/<component name>.cpp

The source file of a single component.

Note that there is a source file for each component. Thus, this chapter may describe more than one file. Also, there always exists a source file

<component name> is the name of the component that the source file belongs to (in the underscore separator notation).

/test

A directory that contains the test for the library.

/test/<test name>.d

The directory for the test with the name <test name>.

Note that there is a directory for each test. Thus, this chapter may describe more than one directory.

This page provides more information on how to write tests and what <test name> stands for in this case.

/test/<test name>.d/resources

optional

A directory that contains various resources that are required by the test. This directory only exists if the test actually requires resource file(s).

This page provides more information on how to write tests and what <test name> stands for in this case.

/test/<test name>.d/resources/(<resource file>)

optional

A resource file for the test.

Note that a test may have multiple resource files. Thus, this chapter may describe more than one file (or none, if the test does not require any resource files).

This page provides more information on how to write tests and what <test name> stands for in this case.

/test/<test name>.d/src

A directory that contains all source files that are required to build the test.

This page provides more information on how to write tests and what <test name> stands for in this case.

/test/<test name>.d/src/test.cpp

The default source file of the test.

This page provides more information on how to write tests and what <test name> stands for in this case.

/test/<test name>.d/src/source.cpp

optional

Additional source file(s) for the test.

Note that a test may have multiple additional source files. Thus, this chapter may describe more than one file (or none, if the test does not require any additional source files). Also, the name source.cpp is only a placeholder and the file(s) can have any name.

This page provides more information on how to write tests and what <test name> stands for in this case.

/.clang-format

The configuration file for clang-format.

/.clang-tidy

The configuration file for clang-tidy.

/.gitattributes

The .gitattributes file.

/.gitignore

The .gitignore file.

/.travis.yml

optional

The configuration file for TravisCI. This file is only present if TravisCI is actually used.

/CMakeLists.txt

The CMake configuration file.

/LICENSE

The license file. Usually, Nostra projects are published under the MIT license.

/README.md

The readme file of the project.

This page provides more information on how to write a readme file.
